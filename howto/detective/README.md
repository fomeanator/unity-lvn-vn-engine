# 🔍 Детектив / расследование

Нелинейное расследование: игрок свободно ходит по локациям, копит улики и предъявляет обвинение, которое засчитывается только при полной картине.

## Что делает пример

Действие в особняке Грейсонов: хозяина нашли мёртвым в кабинете, а игрок — детектив. Есть три места для сбора улик — кабинет, кухня (дворецкий) и гостиная (вдова), — между которыми игрок перемещается через узел-хаб. Каждая локация даёт одну улику, а финальное обвинение «выгорает» только тогда, когда собраны все три. Так нелинейная структура сходится в единственную развязку — `:solved`.

## Структура расследования: хаб + улики

Сердце жанра — **узел-хаб** `:hub`, своего рода «карта расследования». Из него игрок уходит в любую локацию, а каждая локация в конце делает `-> hub` и возвращает его обратно. Это даёт свободный, неупорядоченный обход вместо линейного потока.

Улики копятся в обычном списке: при инициализации `clues = []`, а на месте — `clues = push(clues, "яд")`. Помни, что `push` возвращает **новый** список, поэтому результат присваивается обратно в `clues`.

Прогресс виден игроку всё время — реактивный текст в хабе пересчитывает длину списка:

```
text hud x=4 y=8 size=40 color=#e8e0c8 «Улик собрано: {len(clues)}/3»
```

Повторный осмотр локации защищён проверкой `has`: если улика уже в списке, прыгаем на «пустой» вариант сцены.

```
if has(clues, "яд") -> study_seen
clues = push(clues, "яд")
```

Финальное обвинение гейтится порогом по числу улик — без полной картины дело не закрыть:

```
if len(clues) >= 3 -> solved
Улик мало, обвинение рассыплется в суде. Ищи ещё.
-> hub
```

## Возможности движка, которые тут задействованы

- **Choice-хаб с возвратом** — пункты выбора ведут в локации, каждая локация делает `-> hub`:
  ```
  - 🔍 Осмотреть кабинет -> study
  - 🍷 Допросить дворецкого -> butler
  - 💍 Поговорить с вдовой -> widow
  - ⚖ Назвать убийцу -> accuse
  ```
- **Список улик через `push`/`has`** — накопление состояния и защита от дублей:
  `clues = push(clues, "один бокал")` и `if has(clues, "один бокал") -> butler_seen`.
- **`len` в реактивном тексте** — живой счётчик прогресса: `«Улик собрано: {len(clues)}/3»`.
- **Гейт обвинения** — порог на улики решает исход: `if len(clues) >= 3 -> solved`.

## Разбор по шагам

1. **Инициализация.** В прологе сцены — `clues = []`. Пустой список заводится до первого `push`, чтобы счётчик `{len(clues)}` сразу читался как `0/3`. Дальше ставится фон и нарратив-завязка, после чего `-> hub`.

2. **Хаб с реактивной панелью и пунктами.** Метка `:hub` первой строкой рисует HUD-текст со счётчиком улик, затем строкой нарратива («Гостиная. С чего начнёшь?») держит паузу, под которой висят четыре пункта выбора — три локации и обвинение. Текст HUD пересчитывается при каждом заходе в хаб, поэтому прогресс всегда актуален.

3. **Сбор улики с защитой от повтора.** Каждая локация (`:study`, `:butler`, `:widow`) устроена одинаково: ставит свой фон, даёт реплику/нарратив, а затем `if has(clues, "...") -> ..._seen` — если улика уже собрана, управление уходит на «виденный» узел (`:study_seen` и т.п.), который просто говорит фразу-заглушку и возвращает `-> hub`. Если улики ещё нет — `clues = push(clues, "...")`, подтверждение «[записано]» и `-> hub`. Так одну и ту же улику нельзя засчитать дважды.

4. **Узел `:accuse` с порогом.** Обвинение не ветвит по подозреваемому — оно проверяет полноту улик: `if len(clues) >= 3 -> solved`. Если порог не достигнут, поток падает дальше на строку «Улик мало…» и `-> hub`, отправляя игрока добирать недостающее.

5. **Развязка `:solved`.** Финальный узел сводит цепочку улик в обвинение вдовы и завершает сценарий через `-> __end`. Это единственная «правильная» концовка, в которую сходится весь нелинейный обход.

## Запуск и проверка

```sh
# собрать транскодер
cd tools/lvnconv && go build -o /tmp/lvnconv .

# скомпилировать .lvns → .lvn
/tmp/lvnconv convert -i howto/detective/detective.lvns -o /tmp/det.lvn

# структурная проверка: неизвестный op, висячие прыжки, дубли меток
/tmp/lvnconv validate /tmp/det.lvn
```

Цель — **0 warning(s)**. Каждый «виденный» узел (`:study_seen` и др.) достигается прыжком из `if has(...)`, а перед ним стоит явный `-> hub`, так что ворнинга «reached by fall-through» быть не должно.

## Сделай своим

- **Больше улик и подозреваемых.** Добавь локации и подними порог: `if len(clues) >= 5 -> solved`, а счётчик переведи на `{len(clues)}/5`.
- **Ложные улики.** Клади в список и пустышки, но в гейте проверяй наличие конкретных ключевых через `has`, а не только `len` — тогда лишние улики не помогут.
- **Обвинение конкретного подозреваемого.** Замени единый `:accuse` на выбор «Кого обвиняете?» и в каждой ветке проверяй нужный набор: `if has(clues, "яд") && has(clues, "вино от вдовы") -> solved`, иначе — ложное обвинение и плохая концовка.
- **Время-ходы.** Заведи `moves = 0`, увеличивай на каждом заходе в локацию (`moves = moves + 1`) и закрывай дело по дедлайну: `if moves >= 8 -> too_late`.
- **Допросы с ветвлением.** Внутри `:butler`/`:widow` добавь под-выборы (давить / поверить), которые открывают разные улики или меняют реакцию свидетеля.
